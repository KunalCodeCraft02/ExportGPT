# Plan: Replace Local Image Storage with Cloudinary

## Problem
Product images are stored locally in `uploads/products/`. Render has no persistent local storage, causing `ENOENT` errors after redeploy/restart.

## Codebase Analysis

### Current Image Flow (traced end-to-end)

```
WhatsApp Image Message
  → webhook.controller.js:handleWebhook (line 110-116)
    → getMediaUrl(mediaId) (line 912-947)
      → Facebook Graph API: GET /{mediaId} → gets temporary URL
      → Downloads image as arraybuffer
      → fs.writeFileSync("uploads/products/{mediaId}.jpg")  ← THE PROBLEM
      → Returns "/uploads/products/{mediaId}.jpg"
    → handleImageMessage → handleProductImageMessage
      → Stores path in productData.images[] and productData.thumbnail
  → finishProductRegistration
    → Product.create({ images, thumbnail })  ← LOCAL PATH SAVED TO MONGO
```

### Files Involved

| File | Role | Lines of Interest |
|------|------|-------------------|
| `src/controllers/webhook.controller.js` | Downloads WhatsApp media, saves to disk | L912-947 (`getMediaUrl`), L110-116 (image handling), L1371 (delete product) |
| `src/server.js` | Serves `/uploads` as static files | L23-24 |
| `src/models/Product.js` | Stores `images: [String]`, `thumbnail: String` | L25-26 |
| `src/services/product.service.js` | Product CRUD, image step handling | L100-115 (image message), L129-163 (finish registration), L422-424 (delete) |
| `views/admin/products.ejs` | Displays product images in admin panel | L59-63 |
| `src/routes/admin.routes.js` | Admin routes for product delete | L45 |

### Key Findings
1. **No multer** - Images come from WhatsApp API, not HTTP file uploads
2. **Single disk write point** - Only `getMediaUrl()` in webhook.controller.js writes files
3. **No image cleanup** - Product delete removes MongoDB doc but never deletes the image file
4. **No temp file cleanup** - If process crashes after write but before response, orphan files accumulate
5. **Admin panel** - Uses `product.thumbnail || product.images[0]` which works with any URL

## Implementation Plan

### Step 1: Install Cloudinary & Create Config

**New file: `src/config/cloudinary.js`**
- Import and configure cloudinary v2
- Read `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` from env
- Export configured instance
- Validate all three env vars exist on startup, log warning if missing

**Edit: `package.json`**
- Add `"cloudinary": "^2.5.1"` to dependencies

**Edit: `.env`**
- Add placeholder entries for the three Cloudinary env vars

### Step 2: Create Cloudinary Upload Helper

**New file: `src/services/cloudinary.service.js`**
- `uploadImage(buffer, options)` → uploads buffer to Cloudinary folder `exportconnect/products`, returns `{ secure_url, public_id }`
- `deleteImage(publicId)` → deletes image from Cloudinary by public_id
- `deleteImages(publicIds[])` → batch delete, never throws (best-effort)
- All functions use `uploadstream` or `upload` with `resource_type: "image"`
- Wrap Cloudinary calls in try/catch, log errors, re-throw with meaningful messages

### Step 3: Replace `getMediaUrl()` in webhook.controller.js

**Edit: `src/controllers/webhook.controller.js`**

Replace the current `getMediaUrl()` (lines 912-947) which:
- Downloads image to buffer (keep this)
- Writes to `uploads/products/` with fs (REMOVE)
- Returns local path (CHANGE to return Cloudinary URL)

New flow:
```
getMediaUrl(mediaId):
  1. Facebook API → get temporary URL (unchanged)
  2. Download image as arraybuffer (unchanged)
  3. Write buffer to a temp file in uploads/products/ (TEMPORARY)
  4. Upload temp file to Cloudinary via cloudinary.service.js
  5. Delete temp file in finally block (always, even on error)
  6. Return { url: secure_url, publicId: public_id }
```

Update `handleImageMessage()` (line 902-910) to pass both `url` and `publicId` to `handleProductImageMessage`.

**Why temp file is needed**: Cloudinary's `upload` accepts file paths or buffers. We can use buffer directly with `upload_stream` to avoid any temp file entirely. Decision: use `upload_stream` with the buffer — no temp file needed.

Revised new flow:
```
getMediaUrl(mediaId):
  1. Facebook API → get temporary URL (unchanged)
  2. Download image as arraybuffer (unchanged)
  3. Upload buffer directly to Cloudinary via upload_stream
  4. Return { url: secure_url, public_id: public_id }
```

### Step 4: Update Product Schema

**Edit: `src/models/Product.js`**
- Add `imagePublicIds: { type: [String], default: [] }` field (parallel to `images`)
- Keep `images` field (now stores Cloudinary URLs instead of local paths)
- Keep `thumbnail` field (now stores Cloudinary URL)
- Add `thumbnailPublicId: { type: String, trim: true }` for the thumbnail's Cloudinary ID

### Step 5: Update Product Service - Image Handling

**Edit: `src/services/product.service.js`**

Update `handleProductImageMessage(phone, imageUrl, state)` (line 100-115):
- Now receives `{ url, publicId }` instead of just `imageUrl`
- Store `url` in `productData.images[]`
- Store `publicId` in `productData.imagePublicIds[]`
- Store `url` in `productData.thumbnail`
- Store `publicId` in `productData.thumbnailPublicId`

Update `finishProductRegistration(phone, role, tempData)` (line 129-163):
- Pass `imagePublicIds` and `thumbnailPublicId` to `Product.create()`

### Step 6: Update Product Delete - Cloudinary Cleanup

**Edit: `src/services/product.service.js`**

Update `deleteProduct(productId)` (line 422-424):
- Before deleting from MongoDB, fetch the product to get `imagePublicIds` and `thumbnailPublicId`
- Call `cloudinary.service.deleteImages()` with all public IDs
- Then delete from MongoDB

**Edit: `src/controllers/webhook.controller.js`**

Update `handleProductDeleteIntent` (line 1339-1375):
- Currently does `Product.findByIdAndDelete(product._id)` directly
- Change to call `deleteProduct()` from product.service.js which handles Cloudinary cleanup

### Step 7: Update Product Edit - Image Replacement

**Edit: `src/controllers/webhook.controller.js`**

The current edit flow (`handleProductEditIntent`, line 1288-1337) only supports editing price, quantity, description — not images. This is fine. The image is set at creation time. No changes needed for edit unless we want to add image editing later.

### Step 8: Update Admin Panel

**Edit: `views/admin/products.ejs`** (line 59-63)
- Current: `<img src="<%= product.thumbnail || product.images[0] %>">`
- This already works with Cloudinary URLs since they are just URLs
- No changes needed to the template

### Step 9: Update Server.js

**Edit: `src/server.js`**
- Remove `app.use("/uploads", express.static(uploadsPath))` (line 23-24)
- Remove the `uploadsPath` variable (line 23)
- Keep the `uploads/products/` directory for now but it will no longer be used

### Step 10: Environment Variables

**Edit: `.env`** (and document for Render)
```
CLOUDINARY_CLOUD_NAME=your_cloud_name
CLOUDINARY_API_KEY=your_api_key
CLOUDINARY_API_SECRET=your_api_secret
```

### Step 11: Error Handling

In `src/services/cloudinary.service.js`:
- Missing credentials → throw on startup with clear message
- Upload failure → throw with Cloudinary error message
- Network timeout → retry once, then throw
- Invalid image → Cloudinary rejects it, propagate error

In `src/controllers/webhook.controller.js`:
- If Cloudinary upload fails, send user-friendly error message
- Log full error for debugging
- Never leave orphan temp files (use try/finally or upload_stream)
